# Changelog

Хронология изменений **Syrax** — платформы, на которой работает трекинг-бот бегового клуба SHU RUN.

## 1.3.6 — 21 июля 2026 · Технический релиз

**Админ-панель**<br>
— В списке записавшихся на событие добавлен отдельный блок листа ожидания.<br>
— Черновики многошаговых сценариев переживают перезапуск бота: создание события, городская рассылка, оповещение записавшихся, анонсы и сообщения по забегам.<br>
— Команда /cancel полностью сбрасывает сохранённый черновик.<br>

**Рассылки**<br>
— Итоговые отчёты повторяются ограниченное число раз с экспоненциальной паузой и учётом Telegram retry_after.<br>
— После исчерпания попыток отчёт получает терминальный статус и больше не блокирует очередь остальных заданий.<br>
— Рассылки по забегам требуют второго подтверждения — как городские.<br>
— Удаление аккаунта администратора обезличивает созданные им рассылки вместо их удаления: история доставки другим пользователям сохраняется.<br>

**Надёжность**<br>
— Вечерние запросы посещаемости учитывают время города, доступность получателя и постоянные ошибки доставки; недоступные пользователи больше не блокируют очередь.<br>
— Порядок блокировок при истечении предложений листа ожидания приведён к общесистемному.<br>
— Неактивные аккаунты реактивируются только командой /start, поэтому сообщение «Аккаунт неактивен» соответствует фактическому поведению.<br>
— Счётчики ошибок в /health больше не занижаются после накопления 30 различных ошибок.<br>
— Журнал ошибок показывает фактический тип исключения вместо служебной строки.<br>
— Метка стартового алерта перенесена в systemd StateDirectory вместо каталога деплоя, доступного только на чтение, — троттлинг уведомлений при краш-лупе работает корректно.<br>

**Приватность и безопасность**<br>
— Вечерние запросы посещаемости больше не отправляются пользователям, отозвавшим согласие или деактивированным: проверка допуска добавлена в очередь уведомлений.<br>
— Правила регистрации, допуска пользователей к рассылкам и определения завершившихся событий приведены к единым источникам.<br>
— DB_SSL_DISABLED=1 действительно отключает вложенный MySQL TLS внутри SSH-туннеля.<br>

**Архитектура и производительность**<br>
— Инфраструктурные модули перенесены в core/, независимые бизнес-сервисы — в services/.<br>
— Реализации общих UI-, parsing- и audit-хелперов централизованы; в отдельных модулях оставлены только тонкие адаптеры.<br>
— Лестница автоматических наград вынесена в единый источник данных.<br>
— Profile API переведён с устаревшего startup-хендлера FastAPI на lifespan.<br>
— Сокращено количество повторных запросов к БД при показе карточки события, отметке посещения и маршрутизации текстовых сообщений.<br>
— Выборки событий по месяцам используют полуоткрытый диапазон дат вместо DATE_FORMAT, что позволяет MySQL использовать индекс по дате события.<br>
— Runtime fingerprint больше не включает файлы виртуальных окружений.<br>

**Схема базы и обновление**<br>
— Миграция 009: добавлена таблица conversation_states, поле broadcast_jobs.admin_chat_id стало nullable; schema contract V5.<br>
— Миграция 010: добавлены report_attempts, next_report_at, report_failed_at и индекс очереди отчётов; schema contract V6.<br>
— Миграции применяются до перезапуска сервисов: бот и API не запускаются на неприменённой схеме, поэтому работа с несовместимой базой невозможна.<br>
— В релизе меняется Mini App, поэтому деплою нужно явно указать проверенный web-root.<br>
— Unit shurunbot-prod.service получил StateDirectory=shurunbot; deploy-скрипт переустанавливает unit автоматически.<br>
— История ветки перед релизом была переписана: три коммита про SSH-туннель схлопнуты в один без изменения содержимого. Существующим рабочим копиям нужен git fetch origin и git reset --hard origin/main.<br>

**Разработка**<br>
— Добавлен скрипт синхронного бампа всех cache-busting версий Mini App с проверкой маркеров до записи файлов.<br>
— Набор расширен с 231 до 267 автоматических тестов.<br>
— Обновлена документация в CLAUDE.md.<br>

## 1.3.5 — 21 июля 2026 · Релиз безопасности

**Защищённое подключение к MySQL**<br>
— Production переведён с прямого незашифрованного подключения к Beget MySQL на постоянный SSH-туннель.<br>
— API и бот подключаются к локальному адресу 127.0.0.1:13306, откуда трафик передаётся внутри SSH на MySQL в Beget.<br>
— Production-процессы больше не используют прямое внешнее подключение к Beget на порту 3306.<br>
— Для локального подключения настроен отдельный пароль MySQL-доступа localhost.<br>
— Работа Profile API, Telegram-бота, фоновых воркеров и резервного копирования через новый маршрут проверена на production.<br>

**Изоляция туннеля**<br>
— Добавлен отдельный systemd-сервис shurunbot-mysql-tunnel.service.<br>
— Туннель запускается от отдельного непривилегированного пользователя shurun_tunnel.<br>
— Закрытый SSH-ключ туннеля недоступен runtime-пользователю Telegram-бота.<br>
— Для ключа запрещены PTY, X11, agent forwarding и выполнение пользовательского SSH rc.<br>
— Перенаправление разрешено только к MySQL на 127.0.0.1:3306; произвольные SSH-туннели запрещены.<br>
— Туннель слушает только локальный интерфейс и не открывает MySQL-порт наружу.<br>
— Добавлены строгая проверка host key, keepalive и автоматическое восстановление соединения.<br>
— Tunnel-service включён в автозагрузку и автоматически запускается после перезагрузки сервера.<br>

**Fail-closed запуск**<br>
— Systemd-сервисы Profile API и Telegram-бота требуют активный MySQL-туннель перед запуском.<br>
— При недоступном туннеле приложения не переключаются на прямое незашифрованное соединение.<br>
— В production .env закреплено подключение через 127.0.0.1:13306.<br>

**Проверка перед деплоем**<br>
— Deploy проверяет корректность tunnel-настроек production .env.<br>
— До остановки сервисов выполняется реальный SELECT 1 через SSH-туннель.<br>
— DB preflight использует тот же systemd EnvironmentFile, что Profile API и Telegram-бот.<br>
— Неверный пароль, неработающий туннель или неправильный порт останавливают релиз до downtime.<br>
— Исправлена изоляция deploy-тестов: они больше не обращаются к настоящему /etc/systemd.<br>

**Проверка production**<br>
— Profile API успешно подключён к MySQL через 127.0.0.1:13306.<br>
— Health-check бота подтверждает доступность БД и работу всех фоновых воркеров.<br>
— После запуска не зафиксировано сетевых ошибок Telegram и ошибок приложения.<br>
— Локальная резервная копия успешно создана через SSH-туннель и загружена в Yandex Object Storage.<br>
— Набор расширен с 227 до 231 автоматического теста.<br>
— Пользовательские функции, схема базы, зависимости и Mini App в этой версии не менялись.<br>
— Fingerprint после обновления не изменился: sha256:1d25dd061165, поскольку пользовательский backend-код не менялся.<br>

## 1.3.4 — 21 июля 2026 · Инфраструктурный релиз

**Production-деплой**<br>
— Добавлен fail-closed сценарий обновления production с обязательным указанием ожидаемого commit или tag.<br>
— Перед обновлением проверяются чистота Git checkout, свежесть резервной копии и точное совпадение с origin/main.<br>
— Целевая версия тестируется во временном Git worktree до остановки production-сервисов.<br>
— Deploy автоматически определяет изменения миграций, зависимостей, Mini App и systemd-конфигурации.<br>
— Останавливаются и перезапускаются только затронутые сервисы.<br>
— Profile API запускается первым; после проверки /api/health и fingerprint запускается Telegram-бот.<br>
— При ошибке затронутые сервисы остаются в безопасном остановленном состоянии с понятным отчётом.<br>
— Проверка свежести backup выполняется от runtime-пользователя, которому принадлежат резервные копии и служебные marker-файлы.<br>

**Production-конфигурация**<br>
— Systemd-unit-файлы бота и Profile API перенесены в репозиторий.<br>
— Добавлены узкое sudoers-правило для test-синхронизации, пример nginx reverse proxy и пример конфигурации restore-БД.<br>
— Profile API переведён на production virtualenv /opt/shurunbot-prod/.venv.<br>
— Устранена зависимость боевого API от тестовой установки /opt/shurunbot.<br>

**Права доступа на сервере**<br>
— Создан отдельный admin/deploy-пользователь artem_admin.<br>
— Runtime-пользователь shurun_bot лишён неограниченного NOPASSWD:ALL и членства в административной группе.<br>
— За shurun_bot оставлены только узкие права, необходимые для автоматической синхронизации test-базы.<br>
— Ограничены права на production .env и разделены полномочия владельца кода и пользователя, запускающего сервисы.<br>

**Резервное копирование**<br>
— Добавлена автоматическая загрузка резервных копий в закрытый Yandex Object Storage.<br>
— IAM-токен получается из metadata VM отдельно для каждого запуска и существует только в памяти процесса.<br>
— IAM-токен Object Storage не сохраняется в .env, файлах или логах; статические ключи доступа к бакету не используются.<br>
— Для поддержки Object Lock при загрузке передаётся Content-MD5.<br>
— Наличие объекта после загрузки подтверждается отдельным запросом к Object Storage.<br>
— При ошибке облачной загрузки локальная копия сохраняется, а администратору отправляется уведомление.<br>
— Для бакета настроены versioning, Object Lock и серверное шифрование отдельным Yandex KMS-ключом.<br>

**Проверка восстановления**<br>
— Добавлены download_backup_for_restore.sh и restore_drill.py для изолированной проверки облачных копий.<br>
— Короткоживущий IAM-токен передаётся downloader-скрипту только через stdin.<br>
— Восстановление разрешено только в отдельную пустую restore-БД с отдельным пользователем.<br>
— Живая production-БД во время проверки не подключается и не изменяется; восстановление выполняется из ранее созданной копии.<br>
— Проверяются целостность gzip и SQL, миграции, schema contract, основные таблицы и отсутствие ключевых orphan-записей.<br>
— Реальное восстановление из KMS-зашифрованной облачной копии успешно проведено.<br>

**Еженедельный отчёт**<br>
— Формирование отчёта вынесено из main.py в отдельный модуль weekly_digest.py.<br>
— Уточнён расчёт общего числа профилей, завершённых регистраций и недельного прироста.<br>
— Незавершённые регистрации разбиваются по этапам: политика, телефон и город.<br>
— Записи считаются по событиям отчётной недели независимо от даты создания записи.<br>
— Отдельно показываются уникальные участники, отмены и оставшиеся в листе ожидания.<br>
— Посещаемость рассчитывается относительно ожидаемых отметок с разделением на подтверждения, пропуски и отсутствующие ответы.<br>
— Посещения, добавленные вручную через /mark, выводятся отдельно и не влияют на основной процент.<br>
— Добавлена единая разбивка событий, записей и отметок по городам.<br>
— Запросы и форматирование отчёта тестируются без Telegram и реальной БД.<br>

**Разработка**<br>
— Добавлены автоматические тесты production-деплоя, резервного копирования, restore drill, конфигурации развёртывания и недельного отчёта.<br>
— Набор расширен с 199 до 227 автоматических тестов.<br>
— Обновлены CLAUDE.md, deploy/README.md и документация тестов.<br>
— Новых миграций базы, зависимостей и изменений Mini App нет.<br>

## 1.3.2 — 15 июля 2026 · Технический апдейт без новых пользовательских функций

**Приватность и безопасность**<br>
— При синхронизации тестовая база больше не получает персональные данные production-пользователей: телефоны, Telegram ID, комментарии и админ-журналы не переносятся, а для тестов создаются синтетические профили.<br>
— Ключ шифрования комментариев стал обязательным и больше не выводится из токена бота или реквизитов базы; добавлена ротация с чтением прежних ключей.<br>
— Загрузка изображений наград ограничена списком доверенных HTTPS-хостов, с проверкой типа файла, лимитом размера и перепроверкой каждого редиректа.<br>
— Перед отправкой рассылки повторно проверяются активность, регистрация и согласие пользователя.<br>

**Рассылки**<br>
— Массовые рассылки переведены в долговечную очередь: состояние хранится в базе и переживает перезапуск.<br>
— Повторное нажатие не создаёт дубль рассылки; временные ошибки повторяются ограниченное число раз.<br>
— История рассылок автоматически очищается по истечении срока хранения.<br>

**Исправлено**<br>
— Лист ожидания больше не останавливается после освобождения места.<br>
— Просроченное предложение места нельзя принять, даже если фоновый обработчик ещё не дошёл до него; после завершения события предложения не создаются.<br>
— Старые кнопки больше не позволяют записаться на прошедший забег.<br>
— Личный рекорд не перезаписывается более медленным результатом.<br>
— Прошедшие и будущие забеги в Mini App определяются по дате, а не по наличию результата.<br>
— Удалённая дистанция забега больше не возвращается из старого поля.<br>
— Награду нельзя выдать несуществующему пользователю, а параллельные запросы не создают дубли.<br>
— Нереалистичные дистанции и результаты (включая `NaN` и `Infinity`) отклоняются проверкой.<br>

**Другое**<br>
— Запись на события другого города: через смену города, прямую ссылку на событие или кнопку в рассылке.<br>
— Ключевые правила вынесены в единые сервисы (конфигурация, время, состояние аккаунта, справочники, удаление профиля, отправка сообщений), чтобы бот и Mini App API применяли их одинаково.<br>
— Время: хранение в UTC, бизнес-логика в московском времени.<br>
— Дистанции забегов хранятся в одном каноническом источнике.<br>
— Добавлены миграции 004–008; при старте проверяется фактическая структура базы, а все таблицы актуальной схемы приложения приведены к utf8mb4.<br>
— Проверка фотографии Telegram-профиля кэшируется.<br>

## 1.3.1 — 15 июля 2026 · Технический апдейт без новых пользовательских функций

**Надёжность базы данных**<br>
— Изменения структуры БД вынесены в пронумерованные миграции: они применяются отдельно перед запуском сервисов.<br>
— Бот и Mini App API больше не изменяют таблицы при старте и не скрывают ошибки схемы.<br>
— Если схема БД не соответствует версии кода, сервис не запустится до успешного применения миграций.<br>
— Защита от одновременного применения миграций к одной базе — два процесса не мигрируют её параллельно.<br>

**Стабильность бота**<br>
— Временный сбой Telegram при нажатии inline-кнопки больше не отменяет запись, отмену, навигацию или действие администратора.<br>
— Тайм-ауты Telegram внутри callback-обработчиков корректно отображаются в `/health` как сетевые ошибки, а не как ошибки приложения.<br>

**Мониторинг**<br>
— API Mini App получил полноценный `/api/health`: проверка доступности БД, задержка ответа, версия кода, время запуска и uptime процесса.<br>
— Добавлен контроль резервных копий: супер-админы получают уведомление при ошибке бэкапа, слишком маленьком, отсутствующем или устаревшем дампе, а также при ошибке синхронизации тестовой БД.<br>
— Добавлена независимая почасовая проверка свежести резервной копии.<br>

## 1.2 — 12 июля 2026

**Главное**<br>
— Общая логика записи в боте и мини-аппе вынесена в единый модуль — оба входа работают по одним правилам.<br>
— Исправлено превышение лимита мест при одновременной записи нескольких человек.<br>
— 36 автотестов записи, очереди и аварийных сценариев — прогоняются без основной БД.<br>

**Лист ожидания**<br>
— Место, предложенное участнику очереди, считается временно занятым: его нельзя перехватить обычной записью, а два предложения на одно место больше не выдаются.<br>
— Если сообщение с предложением места не доставлено, участник сразу возвращается в лист ожидания.<br>
— Если бот перезапустился между созданием предложения и отправкой сообщения, потерянное состояние обнаруживается автоматически: участник возвращается в очередь, после чего место предлагается повторно.<br>
— Добавлена полная сверка очереди после запуска бота и затем каждые 15 минут. Несогласованные состояния исправляются автоматически, а супер-администраторы получают сводку об исправлениях.<br>

**Диагностика**<br>
— Команда /health для супер-админов: доступность БД, состояние фоновых воркеров, зависшие предложения мест, журнал последних ошибок и версия запущенного кода.<br>

**Мини-апп**<br>
— Блок «Ближайшая тренировка» после 21:30 МСК больше не показывает сегодняшнее событие, запись на которое уже закрыта.<br>

## 1.0 — 7 июля 2026 · Релиз

**Главное**<br>
— Бот работает в четырёх городах: Санкт-Петербург, Москва, Екатеринбург и Казань.<br>
— При регистрации пользователь выбирает город, в котором тренируется с клубом. Эта настройка влияет на уведомления, которые ему будут приходить. Например, участникам из Москвы — только о московских событиях.<br>
— При регистрации бот просит контактный номер телефона — на случай чрезвычайной ситуации на тренировке. Номер виден только организаторам.<br>
— Город можно сменить в любой момент.<br>
— Возможность оставить заявку на попадание в одну из рубрик в тг-канале.<br>

**Запись на тренировки и другие события**<br>
— Пользователи видят список предстоящих мероприятий и могут на них записаться.<br>
— На отдельной странице показываются актуальные записи, в любой момент можно отказаться от участия.<br>
— У события может быть ограниченное количество мест.<br>
— Если свободных мест нет, пользователь может записаться в лист ожидания.<br>
— Когда кто-то откажется от участия, первому в листе ожидания придёт оповещение с предложением записаться в течение 30 минут.<br>

**Журнал тренировок**<br>
— Если пользователь был записан на клубную тренировку — беговую работу, городскую или длительную пробежку, йогу, ОФП — вечером ему придёт сообщение с просьбой подтвердить, что он приходил.<br>
— Для мероприятий такой запрос включается по желанию организатора.<br>
— Время запроса подстраивается под город и тип: в Екатеринбурге приходит по местному времени.<br>
— После подтверждения участия пользователь может выбрать, насколько тренировка была сложной, и оставить приватный комментарий.<br>
— Можно помечать тренировки, которые уже прошли — если пользователь не был записан, но пришёл.<br>

**Награды**<br>
— Автоматические награды за регулярность тренировок.<br>

**Админ-панель**<br>
— Еженедельная статистика: по понедельникам суперадминам приходит отчёт — сколько всего пользователей, прирост за неделю, разбивка по городам и записи на события.<br>
— Создание, редактирование и удаление событий.<br>
— Рассылки по городам с предпросмотром и добавлением кнопки «Записаться».<br>
— Просмотр списка записавшихся на событие — с контактными номерами для экстренной связи.<br>
— Оповещение участников конкретного события.<br>
— У стандартных типов тренировок — общие названия с эмодзи, при этом каждому событию можно задать своё название под город (например, место старта).<br>
— Изменение событий, оповещения и списки записавшихся сгруппированы по городам.<br>
— Журнал действий: все операции админов (создание, правки, рассылки) записываются в аудит.<br>
— Раздел с заявками пользователей на рубрики канала.<br>

**Обратная связь**<br>
— Через раздел «О боте» можно отправить анонимную обратную связь — сообщения попадают к администраторам.<br>

**Приватность и безопасность**<br>
— При первом запуске пользователь принимает политику конфиденциальности и правила сообщества.<br>
— Приватные комментарии к тренировкам хранятся в зашифрованном виде.<br>
— Защита от спама и работа только в личных чатах.<br>
